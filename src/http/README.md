# HTTP Contract Scaffold

This folder contains framework-agnostic HTTP boundary contracts for Swooshz Platform.

- `session-cookie.ts` parses the browser cookie header, extracts the platform session reference, builds session `Set-Cookie` headers, and builds logout clearing cookies.
- `auth-handlers.ts` defines framework-agnostic browser auth start/callback handler contracts over injected OIDC adapter, state store, and callback service dependencies.
- `handlers.ts` adapts plain request-like objects to the storage-agnostic protected app-access, current session context, app launch intent, SQAG browser launch handoff, CSRF token issuance, and session revocation services.
- `platform-shell.ts` renders the framework-free internal landing page and app shell HTML for the Node adapter.
- `route-contracts.ts` records the first approved route map without wiring a real server or framework.
- `origin-validation.ts`, `csrf.ts`, and `request-security.ts` provide framework-agnostic Origin/Referer and CSRF token validation contracts for future state-changing browser-cookie routes.
- `csrf-token-repositories.ts` and `csrf-token-service.ts` define the storage-agnostic CSRF token lifecycle used by repository-backed validators.
- `csrf-token-crypto.ts` provides Node crypto-backed CSRF token generation and HMAC hashing adapters for future live dependency composition.
- `node-adapter.ts` adapts Node-style HTTP request data to the approved route contracts for `GET /`, `GET /app`, `GET /app/admin`, `GET /healthz`, `GET /api/platform/auth/start`, `GET /api/platform/auth/callback`, `GET /api/platform/session/app-access`, `GET /api/platform/session/context`, `GET /api/platform/session/csrf`, protected workspace admin member/app-entitlement/audit routes, `POST /api/platform/apps/launch`, `POST /api/platform/apps/launch/open`, `POST /api/platform/apps/launch/consume`, and `POST /api/platform/logout`.
- `runtime-config.ts` parses privacy-safe Node HTTP runtime config for local and production hosting posture.
- `node-server.ts` creates a Node HTTP server around the adapter only when explicitly invoked.

The cookie value is a platform session reference only. It must not contain provider tokens, auth codes, raw OIDC claims, client secrets, session secrets, database URLs, SQAG workflow data, quote exports, pricing files, embedded logos, or customer/company/bank data.

This scaffold does not implement cryptographic signing. Future real HTTP wiring must use high-entropy opaque session references generated by an approved secure factory and persisted as platform session records. Do not invent weak signing or store raw session tokens in the database.

ADR 0007 defines the HTTP transport and CSRF strategy. State-changing browser-cookie routes must use Origin/Referer validation plus a CSRF token strategy when real route wiring is implemented. Logout is POST-only in the route contract even though the logout response remains idempotent and privacy-safe.

Future real HTTP adapters must call `validateHttpRequestSecurityForRoute` before invoking state-changing browser-cookie handlers.

CSRF token lifecycle logic is now defined behind injected token factory, token hash, and repository boundaries. The service stores only token hashes and metadata; raw tokens may be returned only by issuance responses for an active browser session. CSRF issuance responses set `Cache-Control: no-store, no-cache, must-revalidate`, `Pragma: no-cache`, and `Expires: 0`. Real secure token generation and HMAC hashing adapters exist, and the Drizzle repository adapter stores only token hashes. Live dependency composition and database execution remain deferred.

Runtime dependency composition now lives under `src/runtime`. It assembles the approved Node adapter dependency object from an injected Drizzle database object, runtime config, a CSRF hash secret config, a deterministic `now` factory, and an injected CSRF token id factory. When auth dependencies are supplied, it also reads the existing auth config contract, uses the Drizzle auth state store, secure state/nonce factories, and an `AUTH_STATE_HASH_SECRET`-backed HMAC reference factory. Auth can use an injected OIDC adapter or the explicit `generic_oidc` provider mode, which composes the generic OIDC adapter plus JWT/JWKS verifier with injected HTTP. SQAG browser handoff can use an injected server-side HTTP boundary only when explicitly configured. This composition does not create a DB connection, run migrations, listen on a port, call provider HTTP, or call SQAG.

Explicit Node bootstrap wiring also lives under `src/runtime`. It reads runtime and secret config from injected environment values, creates the DB client through the existing database boundary only during `start()`, composes dependencies, starts the server only through explicit `start()`, and closes the server plus owned DB client on `stop()`. When an OIDC adapter is injected or `PLATFORM_AUTH_PROVIDER_MODE=generic_oidc` is set, bootstrap can include auth start/callback dependencies and requires `AUTH_STATE_HASH_SECRET`; generic mode also requires issuer and JWKS config plus an injected HTTP client. When `PLATFORM_SQAG_LAUNCH_MODE=server_handoff` is set by the start CLI, bootstrap receives a SQAG HTTP client boundary but does not call it until the browser launch route is invoked. Creating the bootstrap object and calling `start()` do not call provider networking or SQAG. It does not run migrations or create SQAG storage.

The Node adapter does not create or listen on a server. It keeps routing limited to the approved manifest, renders `GET /` and `GET /app` as `text/html; charset=utf-8` with no-store headers, delegates app-access, session context, app launch intent, and logout behaviour to existing handlers, and calls `validateHttpRequestSecurityForRoute` before state-changing browser-cookie routes.

`GET /` is the minimal internal Swooshz Platform landing page. It links to `GET /api/platform/auth/start` and does not expose runtime config or secrets.

`GET /app` is the minimal internal platform shell. The page can render without a browser session and uses browser JavaScript to call `GET /api/platform/session/context`, `GET /api/platform/session/csrf`, `POST /api/platform/apps/launch/open?workspaceId=...&appKey=sqag`, and `POST /api/platform/logout`. It displays the current safe user/workspace/app access summaries returned by the existing JSON API and only shows launch buttons when app access is allowed. SQAG browser launch still requires the browser session cookie plus Origin/Referer and CSRF validation through the existing handler path. The shell does not render, copy, store, or place raw launch tokens in browser response bodies, browser storage, or URLs. It does not bypass service contracts or render session cookies, CSRF secrets, token hashes, auth code/state/nonce material, provider tokens, raw claims, database URLs, or private provider responses.

`GET /app/admin` is the minimal internal-alpha workspace admin shell. The page can render without a browser session and uses browser JavaScript to call the safe session context route before choosing an owner/admin workspace. It then calls protected workspace admin routes to list members, add an existing active provider-backed Platform user by email, change roles, disable and reactivate non-owner memberships, list SQAG app entitlement state, enable or disable the SQAG entitlement, and list recent workspace audit events for the Activity section. State-changing admin actions fetch a CSRF token from `GET /api/platform/session/csrf`, send it as `x-csrf-token`, and rely on the Node adapter's Origin/Referer plus CSRF checks before mutation. The add-existing-user path is an internal-alpha fallback only; it does not deliver invitations, create provider users, create provider identities, auto-reactivate disabled memberships, expose private provider identity internals, or touch SQAG quote data.

`GET /api/platform/workspaces/:workspaceId/audit-events` is the read-only owner/admin workspace audit browsing route. It requires the browser session cookie, does not require CSRF, accepts optional `limit`, and returns no-store safe audit summaries sorted newest first. The handler calls `listWorkspaceAuditEventsForAdmin`, which reuses the same owner/admin context checks as the mutation services and returns only event ids, workspace ids, actor user ids, event/target ids, timestamps, and allowlisted primitive metadata. It does not expose provider identities, private auth claim material, session ids, tokens, cookies, database internals, or SQAG quote data. Export, filtering, and retention workflows remain future scope.

`GET /api/platform/session/context` is a read-only browser-session endpoint for future platform shell/dashboard use. It requires the platform session cookie, does not require CSRF, and returns no-store/no-cache responses containing only safe active-session status, platform user summary, accessible workspace memberships, and app access summaries derived from existing app entitlement rules. It does not expose raw cookies, provider subjects, provider tokens, raw claims, state/nonce data, CSRF material, DB details, launch tokens, SQAG runtime data, or private app payloads. It does not launch apps or persist workspace selection.

`POST /api/platform/apps/launch?workspaceId=...&appKey=...` is disabled for direct browser raw-token delivery. It requires the platform session cookie, Origin/Referer validation, and a CSRF token before returning the safe disabled response. It does not mint or return raw launch tokens to the browser.

`POST /api/platform/apps/launch/open?workspaceId=...&appKey=sqag` is the narrow browser-safe SQAG local UAT handoff. It requires the platform session cookie, Origin/Referer validation, CSRF, explicit `PLATFORM_SQAG_LAUNCH_MODE=server_handoff`, and a configured `PLATFORM_SQAG_APP_BASE_URL`. It creates a normal launch intent server-side, forwards the raw token to SQAG only as `x-app-launch-token` on `POST <sqag-base-url>/api/platform/launch`, returns SQAG's session cookie, and returns only a safe SQAG launch URL to the browser. It fails closed when the SQAG URL host does not match the Platform request host, so it is not an open proxy and not a cross-domain cookie bridge.

`POST /api/platform/apps/launch/consume?appKey=...` is the platform-side app launch token consume endpoint. It requires no browser session cookie and no CSRF token because the raw launch token is the credential. The raw token is accepted only through the `x-app-launch-token` header, never query string. The handler hashes the token before lookup, rejects unknown/expired/consumed/revoked tokens safely, re-checks app access, marks an unconsumed token consumed, and returns no-store/no-cache safe user/workspace/app context. Runtime/bootstrap composition requires `APP_LAUNCH_TOKEN_HASH_SECRET` for both issuing and consuming. These endpoints do not redirect to apps, call SQAG, run migrations, or add app/provider SDKs.

Browser auth route contracts now exist for starting provider login and receiving the provider callback. Auth start stores only hashed state/nonce references through an injected state store and redirects through an injected OIDC authorization URL builder. Auth callback wraps the existing callback service, sets the platform browser session cookie only after successful platform session creation, and keeps provider tokens, raw claims, auth codes, raw state/nonce, state hashes, nonce hashes, and client secrets out of response bodies and errors. Provider networking remains behind the injected OIDC adapter boundary; auth SDKs, frontend login UI, and live state persistence remain deferred.

Auth state lifecycle now has real storage and crypto adapters. `src/auth/auth-state-crypto.ts` provides secure CSPRNG state/nonce factories and HMAC reference hashing with an injected secret, while `src/db/auth-state-repository.ts` stores and consumes only hashed state/nonce references through Drizzle. The generated `auth_states` migration is review-only until an explicit migration execution workflow is invoked.

The Node server factory does not listen on import and does not create live repositories. Production runtime config requires secure cookies, a valid public base URL, and explicit allowed origins. Auth runtime composition requires `AUTH_STATE_HASH_SECRET` only when auth routes are enabled through an injected OIDC adapter or the explicit generic OIDC mode. The OIDC adapter boundary remains the only provider seam; provider SDKs, frontend login UI, billing, deployment, live DB migration execution, and CLI auto-start remain out of scope.

`src/auth/generic-oidc-provider-adapter.ts` provides a provider-agnostic OIDC adapter factory for that seam. It does not run provider discovery, token exchange, userinfo requests, or verification on import, factory creation, runtime composition, or bootstrap start. Token exchange happens only when `exchangeCodeForTokens` is called, and optional userinfo happens only from `verifyTokens` after an injected verifier succeeds. `src/auth/generic-oidc-jwks-verifier.ts` provides a generic JWT/JWKS verifier for the existing verifier interface. It performs real RS256 signature verification with provider JWKS, requires issuer and JWKS URLs, validates issuer, audience, expiry, not-before, issued-at, and subject claims, returns nonce for the adapter-side nonce hash comparison, and keeps nested/raw claim objects out of returned metadata. JWKS fetching happens only during explicit `verify(...)` calls through the injected HTTP boundary. Raw auth codes, client secrets, access tokens, refresh tokens, ID tokens, token responses, JWKS bodies, and raw claims stay out of adapter and verifier return values and errors.

No deployment script, framework route, middleware, SQAG-owned auth, SQAG storage, polished dashboard, broad app proxy, or migration execution is included here.
